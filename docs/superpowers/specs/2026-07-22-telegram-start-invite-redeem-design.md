# Telegram `/start` invite redeem fix — Design

**Date:** 2026-07-22
**Status:** Approved

## Problem

Invite redemption is broken. The invited user opens `t.me/<bot>?start=<code>`, Telegram sends `/start <code>` with a `bot_command` entity, and the `@chat-adapter/telegram` adapter diverts it to the Chat SDK's `processSlashCommand` path (`handleSlashCommandUpdate` returns `true`, so `handleIncomingMessageUpdate` never runs). The message therefore never reaches `onDirectMessage`, where the access gate (`src/mastra/lib/telegram-gate.ts`) expects to parse it. With no slash-command handler registered on the Chat SDK, the event dies silently: the invite is never redeemed, the telegram id is never linked, and the invitee gets no reply.

The supervisor instruction about welcoming users whose message is `\start <code>` (note the backslash typo) never fires for the same reason.

The repository logic (`inviteRepository.redeem`, `userRepository.linkTelegramId`) is correct and unchanged by this design — it simply never executes.

## Fix architecture

### 1. New `/start` handler — `src/mastra/lib/telegram-start.ts`

Registered through the Chat SDK's public API: `agent.channels.sdk.onSlashCommand('/start', handler)`. The `SlashCommandEvent` provides:

- `user.userId` — the sender's telegram id
- `text` — the argument after `/start` (the invite code), possibly empty
- `channel` — handle to post replies into the chat

Behavior, in order:

1. **Known user** (`userRepository.findByTelegramId` hits): post a short fixed greeting ("¡Hola de nuevo! Contame en qué te ayudo."). Telegram sends `/start` every time someone reopens the bot, so silence would read as the bot being dead.
2. **Unknown user with a code**: call `inviteRepository.redeem(code, telegramId)` (atomic, single-use, expiry-checked — existing logic). On success, `userRepository.linkTelegramId(invite.email, telegramId)`, then post the fixed welcome: greet by name when the invite carried `name`, explain in two lines what the bot does, and ask for their name only when we do not have one.
3. **Unknown user with an invalid/expired/missing code**: post a fixed generic message ("No tengo una invitación válida para vos; pedile un link nuevo a quien te invitó."). No token spend, and it does not disclose whether the code ever existed.

The handler follows the same injectable-deps pattern as `createTelegramGate` (a `TelegramStartDeps` type with repository-backed defaults) so it can be unit-tested without Mongo.

All work is wrapped in try/catch with a `console.error`; a Mongo failure must not take down the webhook.

### 2. Simplified gate — `src/mastra/lib/telegram-gate.ts`

The gate no longer parses `/start` — slash commands never reach it. `parseStartCode` and the redeem path are deleted. The gate becomes a pure access check: known senders pass to `defaultHandler`, unknown senders are ignored.

### 3. Supervisor instructions — `src/mastra/agents/mostro-supervisor.ts`

- Delete the `\start <code>` welcome rule (it never fired; the fixed welcome message replaces it).
- Replace the name rule with: if the user introduces themselves or states their name (e.g. replying to the welcome message), save it with `setMyNameTool`.

### 4. Registration point — `src/mastra/index.ts`

Register the handler after the Mastra instance is created.

**Known risk to verify during implementation:** `agent.channels.sdk` is `null` until Mastra initializes the agent's channels ("after the server is ready" per `AgentChannels.initialize`). The implementation must verify the correct registration timing (possibly deferred/retried). This is the only point of technical uncertainty.

## Error handling

- Handler-level try/catch, logged with a `[telegram-start]` prefix.
- `redeem` returning `null` (used, expired, or nonexistent code) and `linkTelegramId` returning `false` (no user for the invite email) fall through to the generic invalid-invite message, with a `console.warn` for the link failure (mirrors the current gate's warn).

## Testing

- **Unit tests** for the new handler with injected deps (same style as `tests/telegram-gate.test.ts`): known user, valid code, invalid code, expired code, missing code, concurrent double-redeem (one wins).
- **Adjust** `tests/telegram-gate.test.ts` (redeem path removed) and `tests/invites.test.ts` if it references `parseStartCode`.
- **Manual end-to-end** check with the real bot (via ngrok) at the end: open an invite link, verify redemption, linking, and the welcome message.

## Out of scope

- Invite listing/revocation, resend flows, or changes to invite creation.
- Welcome message generated by the LLM (rejected: requires replicating the channels pipeline's thread mapping by hand).
